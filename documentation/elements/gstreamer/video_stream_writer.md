# VideoStreamWriter

## Overview

**`VideoStreamWriter`** is a **legacy** (pre-PipelineElement) wrapper
that accepts frame dictionaries on a bounded queue, encodes them to
H.264 and transmits them over the network — by default as **RTP over
UDP** (`rtph264pay ! udpsink`, the pairing consumed by
[VideoStreamReader](video_stream_reader.md) in `--RTP` mode), or, when
an `rtmp_url` is supplied, muxed to FLV and pushed to an **RTMP**
server (e.g. a WebRTC/streaming gateway).

There is no current-style equivalent yet — RTSP *serving* is the
work-in-progress `create_targets()` half of
[DataSchemeRTSP](scheme_rtsp.md); this wrapper remains the only working
network video output in the repository.

**Why you'd use it**: the output half of a legacy chain, e.g. relaying
a camera to another host via [video_example](video_example.md):

```python
from aiko_services.elements.gstreamer import VideoStreamWriter
writer = VideoStreamWriter("192.168.1.155", "5000", 640, 480, "25/1")
writer.write_frame({"type": "image", "id": 1, "image": ndarray})
```

The raw GStreamer RTMP shape it automates (from the usage header):

```bash
gst-launch-1.0 videotestsrc ! \
  video/x-raw,width=852,height=480,framerate=\(fraction\)24/1 ! \
  x264enc pass=pass1 threads=0 bitrate=900 tune=zerolatency ! \
  flvmux name=mux ! rtmpsink location="rtmp://…/WebRTCAppEE/…"
```

## For application developers

### Command-line usage

No `__main__` of its own — exercised via the
[video_example](video_example.md) CLI (`-os output_hostname:port`; the
`rtmp_url` mode is not reachable from that CLI):

```bash
./video_example.py -if ../data/football_test_0.mp4 -os 192.168.1.65:5000 \
  -r 1280 720 -f 30/1 -cv
```

### Public API

```python
class VideoStreamWriter:
    def __init__(self, hostname, port, width, height, framerate,
        rtmp_url=None): ...
    def write_frame(self, frame): ...   # enqueue; blocks when queue full
    def queue_size(self): ...
```

- Default (UDP) mode: `appsrc ! videoconvert ! <get_h264_encoder()>
  <get_h264_encoder_options()> ! rtph264pay config-interval=5 pt=96 !
  udpsink host=<hostname> port=<port>` — the platform encoder options
  from [utilities](utilities.md) apply (zero-latency tuning on Linux).
- RTMP mode (`rtmp_url` given): `appsrc ! videoconvert !
  x264enc bitrate=1000 … tune=zerolatency ! h264parse ! queue !
  flvmux streamable=true ! rtmpsink location="<rtmp_url>" sync=false`
  — note `x264enc` is hard-coded here (no Mac OS X `vtenc_h264` path)
  and `hostname` / `port` are ignored.
- `width`, `height`, `framerate` (+ format `RGB`) set the appsrc caps —
  they describe the **incoming** frames; timestamps are generated by
  `do-timestamp=True`.
- `write_frame(frame)` accepts the family frame dicts:
  `{"type": "image", "image": ndarray, ...}` is encoded and sent;
  `{"type": "EOS"}` emits GStreamer end-of-stream. The internal
  `Queue(maxsize=30)` blocks the caller when full.

## For framework developers (internals)

### Design

```
 write_frame() ─► Queue(30) ─► daemon thread _run_consume_queue():
                                 image ─► tobytes() ─► Gst.Buffer
                                       ─► appsrc push-buffer
                                 EOS  ─► appsrc end-of-stream
     appsrc (is-live, do-timestamp, TIME)
       ! videoconvert ! H.264 encode
       ├─ default: rtph264pay ! udpsink host:port     (RTP/UDP)
       └─ rtmp_url: h264parse ! flvmux ! rtmpsink     (RTMP)
```

- Same producer/consumer shape as
  [VideoFileWriter](video_file_writer.md): bounded queue, daemon
  thread, `parse_launch()` pipeline (the To Do asks for a hand-built
  pipeline, as [video_stream_reader](video_stream_reader.md) has).
- A second, **disabled** thread (`_run_constant_framerate`, commented
  out) would re-push the last buffer every 40 ms via the module-level
  `data` global — a keep-alive for receivers that stall without a
  constant frame rate. The global and the flake8 `F824` note remain in
  the code.

### Implementation notes

- The last-encoded frame bytes are kept in the module-level global
  `data` (shared by the two thread bodies) — a second
  `VideoStreamWriter` instance in one process would share/overwrite it
  if the keep-alive thread were ever re-enabled.
- No bus watching: encode or network errors are silent; there is no
  clean shutdown (daemon thread, no `close()`).
- The Python 2 `Queue` import fallback is dead code.
- RTP receivers need the stream's caps out-of-band;
  `config-interval=5` re-sends SPS/PPS every 5 s so late joiners can
  sync, matching the fixed caps in
  [VideoStreamReader](video_stream_reader.md)'s `udpsrc`.

## Current limitations and roadmap

From the source To Do list:

- Replace `Gst.parse_launch()` with a hand-built pipeline

Additional observed gaps:

- Constant-framerate keep-alive thread is disabled (commented out) and
  its module-global `data` design would need rework (per-instance
  state, locking) before re-enabling
- RTMP mode hard-codes `x264enc` and `bitrate=1000` — no Mac OS X
  encoder, no bitrate parameter; UDP mode bitrate is likewise not
  configurable beyond `get_h264_encoder_options()`
- `rtmp_url` mode is not reachable from the
  [video_example](video_example.md) CLI
- No error reporting, no clean shutdown, no test coverage
- Long-term: network video output should become a
  [DataTarget](../../concepts/data_source_target.md) behind a
  DataScheme (`rtsp:` serving in [scheme_rtsp](scheme_rtsp.md) is the
  work-in-progress successor)
